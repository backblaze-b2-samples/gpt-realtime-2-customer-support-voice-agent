# Feature: Call Bundles

## Purpose
At end-of-call, persist a complete, self-contained record of the conversation to Backblaze B2 — audio, transcript, tool trace, summary, and a manifest — so operators can review the call later from the Calls explorer and so the bucket accumulates a durable corpus of AI artifacts.

## Used By
- UI: `/call` (end-of-call card), `/calls` (list + detail)
- API: `POST /calls`, `GET /calls`, `GET /calls/{id}`, `GET /calls/{id}/audio`, `DELETE /calls/{id}`

## Core Functions
- `services/api/app/service/call_orchestrator.py` — assembles bundle, generates summary, writes to B2
- `services/api/app/repo/b2_calls.py::put_call_artifact()` — wraps `PutObject` for a single artifact (orchestrator calls it five times per bundle, in the order described below)
- `services/api/app/repo/b2_calls.py::list_call_ids()` — `ListObjectsV2` with `Delimiter="/"` scoped to the `calls/` prefix; returns the set of `<call_id>` values
- `services/api/app/repo/b2_calls.py::get_call_manifest()` — fetch + parse `calls/<id>/manifest.json`
- `services/api/app/repo/b2_calls.py::get_bundle_artifact()` — fetch raw bytes of any artifact under `calls/<id>/`
- `services/api/app/repo/b2_calls.py::get_presigned_audio_url()` — 10-min presigned URL with `response-content-disposition=inline`
- `services/api/app/repo/b2_calls.py::delete_call()` — deletes every object under `calls/<id>/`
- `services/api/app/repo/openai_client.py::generate_summary()` — non-realtime Chat Completions call for the Markdown summary

`b2_calls.py` lives alongside the starter kit's `b2_client.py`; the two share a single boto3 client via `b2_client.get_s3_client()`. Keeping bundle-specific code in its own module is what lets `b2_client.py` stay focused on the generic file-level operations.

## Canonical Files
- Orchestration pattern: `services/api/app/service/call_orchestrator.py`
- B2 bundle data access: `services/api/app/repo/b2_calls.py`

## Bundle Layout

For each call, the orchestrator writes five objects under a per-call prefix:

```
calls/<call_id>/
  audio.wav          (mixed both-sides audio: 16-bit PCM, 24 kHz, mono — caller mic + agent track summed via the Web Audio API and encoded to a canonical RIFF/WAVE on End Call)
  transcript.jsonl   (one JSON object per turn: { speaker, text, started_at, ended_at })
  tools.jsonl        (one JSON object per tool event: { tool, args, result, ok, latency_ms, timestamp })
  summary.md         (Markdown — title, caller intent, resolution, key facts)
  manifest.json      (call_id, started_at, ended_at, duration_seconds, tool_count, deflected, audio_bytes, model)
```

The 24 kHz sample rate matches what the OpenAI Realtime API streams on the model side; we resample the local mic into the same rate so the mixed WAV has a single, uniform timebase. Both inputs are attenuated to 0.7 each before summation to avoid clipping during simultaneous speech. See `apps/web/src/lib/audio-recorder.ts` for the encoder.

`<call_id>` is a ULID minted browser-side at Start Call so the bundle key is stable across retries.

## Inputs
- `POST /calls` body: `{ call_id, started_at, ended_at, transcript, tools, audio_base64 }`
  - `transcript`: array of `TranscriptTurn`
  - `tools`: array of `ToolEvent`
  - `audio_base64`: base64-encoded 24 kHz 16-bit mono WAV with both sides of the conversation mixed (see `apps/web/src/lib/audio-recorder.ts`). Empty string when capture failed — the orchestrator still writes the rest of the bundle.
  - Decoded audio is capped at 64 MiB (`MAX_CALL_AUDIO_BYTES` in `services/api/app/types/calls.py`) so malformed or oversized finalize requests fail before bundle writes.

## Outputs
- `POST /calls` → `Call` (`{ call_id, started_at, ended_at, duration_seconds, tool_count, deflected, summary_line, complete }`, per the Pydantic model in `services/api/app/types/calls.py`)
- Audio playback is served separately by `GET /calls/{id}/audio`, which returns a 10-minute presigned URL with `response-content-disposition=inline` (see `b2_calls.py::get_presigned_audio_url`).
- Side effects: 5 objects written to B2 under `calls/<call_id>/`

## Write Order (load-bearing)

The orchestrator writes in this exact order:

1. `audio.wav`
2. `transcript.jsonl`
3. `tools.jsonl`
4. `summary.md` (generated by OpenAI; if generation fails, file still written with `# Summary unavailable\n<error>`)
5. `manifest.json` — last

**`manifest.json` is the durability signal.** Any bundle missing `manifest.json` is treated as "incomplete" by `list_calls()` and the Calls explorer. Operators can delete incomplete bundles via `DELETE /calls/{id}`.

## Flow
1. Browser ends call → POST in-flight session state to `/calls`.
2. Orchestrator writes the four artifact objects with up-to-3-retry exponential backoff on `PutObject` failures.
3. Orchestrator calls `repo.openai_client.generate_summary(transcript, tools)` to produce the Markdown summary.
4. Orchestrator writes `summary.md`, then `manifest.json`.
5. Returns `Call` to the browser, which updates the UI and links to `/calls/<call_id>`.

## Idempotency

- The browser supplies `call_id`; the same bundle can be re-submitted without creating duplicate keys (S3 overwrites the same object key).
- If the orchestrator crashes between step 2 and step 5, the bundle is "incomplete" but consistent — the next POST with the same `call_id` will overwrite the partial state and finish writing.

## Retention

- v1 retention is "forever" — no lifecycle rules applied.
- Future enhancement: a B2 bucket lifecycle policy that ages out `calls/` bundles after N days (out of scope for v1, see plan §8).

## Edge Cases
- `PutObject` fails repeatedly → bundle left partial, ERROR logged with `call_id` and key; explorer surfaces as "incomplete"
- Summary generation fails → `summary.md` written with an error notice, but `manifest.json` is still written
- `audio_base64` empty → bundle persists with an empty `audio.wav`
- `audio_base64` corrupt → 400 from runtime, no objects written
- `audio_base64` over the decoded size limit → 413 from runtime, no objects written
- Same `call_id` POSTed twice → second POST overwrites; final state is whichever request finished last (idempotent for our purposes)
- Bucket missing → 502, no objects written; user is asked to fix config
